# 03. 管理・コンテンツ仕様書

## 1. 言語管理

管理者が学習対象言語を追加・管理する機能。言語を有効化するまで生徒には非公開。

| 操作 | 説明 |
|---|---|
| 言語追加 | 名前・バージョン・アイコン略称を設定してレコード作成 |
| 有効化 / 無効化 | `is_active` トグルで即時切り替え。無効化しても既存の学習データは保持 |
| ツリー編集へ遷移 | 言語一覧から該当言語のスキルツリー管理画面へ直接遷移 |

---

## 2. スキルツリー管理

言語を選択した上でスキルを追加・編集・依存関係を設定する。

### スキルの設定項目
- スキル名、カテゴリ（基礎 / データ構造 / アルゴリズム / OOP / テスト）
- 最大レベル（デフォルト5）
- 前提スキルと必要レベル（解放条件）
- 解放先スキル（このスキルが条件を満たすと解放されるスキル）
- `template_id`（Phase 2以降：共通スキルテンプレートへの紐付け、Phase 1はNULL）

### Phase 2: AI一括展開フロー
1. AIが複数言語のスキルを横断分析し「このスキルは全言語共通の概念」と判定
2. 管理者ダッシュボードに提案として表示（`ai_skill_suggestions` テーブル）
3. 管理者が承認 → `skill_templates` にレコード作成 → 対象 `tags.template_id` を一括更新
4. 却下した場合はスキルの独立性を維持

## 3. 問題生成・承認フロー

### ステータス遷移
```
AUTO_GENERATED → (管理者テスト実行・編集) → APPROVED
                                           → ARCHIVED（破棄）
```

### 問題の属性
| 属性 | 説明 |
|---|---|
| `language_id` | 対象言語 |
| `tag_id` | 紐付けスキル |
| `difficulty` | 1=入門 / 2=中級 / 3=応用 |
| `judgment_type` | `STDOUT` または `TESTCASE`（後述） |
| `source` | `AI_GENERATED` または `MANUAL` |

### 判定方式（judgment_type）の選択基準

管理者が問題を作成・承認する際に設定する。

| judgment_type | 対象 | 検証方法 |
|---|---|---|
| `STDOUT` | 基礎問題（変数・型・ループ・条件分岐） | ユーザーの標準出力と期待値を Diff 比較 |
| `TESTCASE` | 応用問題（関数・再帰・アルゴリズム） | `assert function(input) == expected` 形式で全テストケースを検証 |

**Phase 2以降の拡張予定（judgment_type の追加）**

| judgment_type | フェーズ | 概要 |
|---|---|---|
| `UNITTEST` | Phase 2 | JUnit / pytest / Jest 等のフレームワークを使ったテストコード実行 |
| `EXCEL_DRIVEN` | Phase 4〜 | Excelファイルで定義した大量入出力パターンによるデータ駆動テスト |
| `UI_TEST` | Phase 4〜 | Selenium / Playwright 等を使ったブラウザUIテスト課題 |

### AI生成バッチ
- 夜間または問題在庫が閾値を下回った時に Gemini（Vertex AI）で自動生成
- 生成直後にシステムが模範解答をサンドボックスで自動実行・検証
- テスト通過したもののみ `AUTO_GENERATED` としてDBに保存、管理者レビュー待ちキューに追加

### 管理者レビュー（問題承認画面）
1. `AUTO_GENERATED` 問題の一覧を確認（言語・スキル・難易度でフィルタ可）
2. 管理画面上で「▶ テスト実行」を押して模範解答がテストケースをパスするか確認
3. 問題文・初期コード・テストケースを必要に応じて編集
4. 「承認」で `APPROVED`（全ユーザーの問題プールに追加）、または「破棄」で `ARCHIVED`

## 4. 実行フロー詳細（コード実行〜判定）

### 非同期実行フロー
```
① フロントエンド: POST /execute（コード・problem_id を送信）
② APIサーバー:    task_id を即時返却（202 Accepted）
               :  Celery キューに実行タスクを積む（Redis）
③ Worker:        Docker コンテナ（gVisor）でコードを実行
               :  リソース制限: タイムアウト2秒 / メモリ128MB / ネットワーク遮断
④ 判定エンジン:   エラー判定 → judgment_type に応じた正誤判定 → EXP計算
               :  結果を Redis / DB に書込
⑤ フロントエンド: GET /tasks/{task_id} を500ms間隔でポーリング（最大10秒）
               :  結果取得後にリザルト画面へ遷移
```

### エラー判定（優先度順）
| 種別 | 検出方法 | ユーザーへの表示 |
|---|---|---|
| コンパイルエラー | exit code ≠ 0 かつ stderr に構文エラー | 日本語メッセージ＋エラー行ハイライト |
| 実行時エラー | 実行中に例外発生（IndexError等） | 日本語メッセージ＋スタックトレース翻訳 |
| タイムアウト | 2秒超過 | 「処理時間が超過しました。無限ループの可能性があります。」 |

### 判定エンジン（エラーなしの場合）
**STDOUT モード**
- Dockerの標準出力文字列と `expected_output` を比較（前後の空白・改行はトリム）
- 一致 → PASS、不一致 → FAIL（期待値と実際の出力の差分を表示）

**TESTCASE モード**
- `test_cases` JSONB の各ケースを順番に実行
- `assert function(input) == expected` 形式でサンドボックス内で検証
- 全件通過 → PASS、失敗したケースの入力・期待値・実際の出力を返却

### EXP計算（判定PASS後）
```
base_exp     = 10（同一タグ・難易度の連続解答で減衰: 100%→50%→20%→固定1pt）
first_try    = +5（submissions テーブルで当該問題の提出回数が1回目か確認）
no_hint      = +5（hint_count = 0 か確認）
clean_code   = +5（Linter 警告0件）
efficiency   = +5（runtime_ms・memory_kb が問題設定の基準値以下）
```

## 5. 運用管理

### クォータ管理
- ユーザーランク別のAIリクエスト回数制限を `user_quotas` テーブルで管理
- 制限超過時はヒント要求を制限し、翌日0時にリセット

### コスト・インフラ監視
- Gemini / GPT-4o APIのトークン消費量・コスト推移の可視化
- Dockerコンテナの稼働状況・タイムアウト発生率のモニタリング
- ユーザーの停滞ポイント（離脱率の高い問題・頻出エラー）の分析
