# プログラミング学習システム 全体仕様書（中長期ロードマップ付き）

## 1. プロジェクトビジョン
「コードが動けば正解」という従来の初心者向け学習から脱却し、バグの特定（リーディング）、テストケースの設計、そしてコードの品質（レビュー対応）まで、**実務で自走できるエンジニアを育成するAI駆動型のOJTシミュレーションプラットフォーム**を構築する。

---

## 2. 中長期ロードマップ

### Phase 1: MVP（コア学習体験とインフラの確率）
* **インフラ・セキュリティ**: Dockerコンテナのウォームスタンバイ化によるゼロレイテンシ実行。リソース（CPU/メモリ/ネットワーク）の厳格な制限。
* **AI品質制御**: AI生成問題のDB保存前のシステム事前検証（プレバリデーション）の必須化。
* **学習進行**: スキルツリー（タグ熟練度）システムの導入。
    * 経験者向けのプレースメントテスト（一括アンロック）。
    * EXP減衰システム（反復学習の形骸化防止、復習は最低1pt保証）。
* **UI/UX**: ドメイン制約（ユーザーの年齢や知識に合わせた問題背景）の導入。クリア者限定の「コミュニティ解答（厳選3件）」の匿名閲覧。
* **運営管理**: 管理者ダッシュボードでのAI生成問題のテスト実行と承認フロー。

### Phase 2: 実務へのブリッジ（リーディングと品質管理）
* **課題の標準化**: コンソール出力（print）ベースから、関数（メソッド）ベースへの課題形式の移行。
* **デバッグUX**: 生のスタックトレースのUI上での翻訳と、エラー行のハイライト。
* **バグハント（リーディング）**:
    * 各ユニットの「関所（マイルストーン）」クリア後に、複合的なバグ修正課題を解放。
    * クリア後はAIによる自動生成で無限に復習可能。
* **擬似PRレビューとリテイク**:
    * テストケース通過後、AIシニアエンジニアによるコードレビュー（命名規則、効率性等）を実施。
    * 指摘事項をユーザーが実際に修正（リテイク）し、再チェックをパスして初めて「完全クリア」とする実務サイクルの導入。

### Phase 3: テスト駆動と設計（フルスタック・エンジニアリング）
* **テスト設計訓練**: ユーザーに実装（ゼロからのコーディング）だけでなく、JUnit等のテストコードを書かせる課題を導入し、学習割合を「実装 50%：テスト/リーディング 50%」へ最適化。
* **AIのコンテキスト保持**: ユーザーの「過去に躓いたエラーの傾向」をプロファイル化し、AIのプロンプトに動的注入するパーソナライズ指導。
* **外部環境連携**: ローカルのVSCode/Eclipseへの環境移行支援、および上流工程（基本・詳細設計書からのシステム構築）のシミュレーション。

## 3. システム・アーキテクチャ要件

* **フロントエンド**: SPA (React / Vue.js)
    * 画面左側にエディタとエラー表示UI、右側にAIチャットの2ペイン構造。
* **バックエンド**: APIサーバー (Spring Boot / Python FastAPI)
    * ユーザー学習ログ、タグの経験値計算、問題プール管理。
* **コード実行環境**: Dockerによるサンドボックス
    * 言語ごとのコンテナプール（ウォームスタンバイ）。完全なネットワーク遮断。
* **AIエンジン**: OpenAI API等
    * 非同期バッチ処理による問題の事前生成・ストック。

## 4. コア機能要件（詳細）

### 4.1 学習実行とAIチューター
* ユーザーの書いたコードはシステム側でテストケース（入力値・期待値）とDiff検証を行い正誤を判定。
* AIは「絶対に直接の正解コードを提示しない」ソクラテスメソッドを徹底。行き詰まり防止のため「段階的ヒント」と「ギブアップ機能」を搭載。

### 4.2 スキルツリーシステム (EXP算出ロジック)
固定のコース制を廃止し、タグごとの熟練度で次の概念をアンロックする。
* **獲得EXP（最大30pt）**:
    * 基本クリア (+10pt, 減衰あり: 100% -> 50% -> 20% -> 1pt)
    * 一発正解 (+5pt)
    * ノーヒント (+5pt)
    * クリーンコード/Linter Warningゼロ (+5pt)
    * 実行効率基準クリア (+5pt)

### 4.3 擬似PRレビューフロー (Phase 2)
1. ユーザー提出 -> システム検証 (Pass)
2. AIによる静的解析＆レビューコメント生成。
3. ユーザーが指摘事項（変数名、ロジック改善）を修正し再提出。
4. AIが修正を確認 -> 完全クリア（次の問題が解放）。

## 5. 運営・管理要件 (管理者ダッシュボード)
* **問題の資産化**: AIが非同期に生成した問題（AUTO_GENERATED）を管理者がテスト実行し、品質を確認した上で承認（APPROVED）するワークフロー。
* **バランス調整**: スキルツリーの依存関係編集、レベルアップに必要なEXPテーブルのチューニング。
* **コスト・インフラ監視**: AI APIのトークン消費量予測、Dockerコンテナの稼働状況およびタイムアウト発生率の監視。
